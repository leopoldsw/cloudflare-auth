# Release Readiness Audit

Date: May 15, 2026

This audit maps `cloudflare_auth_implementation_plan.md` to concrete repo
evidence. It is intentionally not a release approval. Maintainers must replace
the pending evidence and signoff files before public beta or 1.0.

## Objective

Ship a polished open-source Cloudflare Auth repo that can be installed, tested,
deployed, consumed as packages, and released only after package ownership,
alpha/beta, deploy-button, and security-review evidence is recorded.

## Non-Negotiable Rules Audit

The implementation plan's numbered rules are audited separately because they are
ship blockers even when a stage-level checklist is green.

| Rule | Requirement                                                                             | Concrete evidence                                                                                                                                                                   |
| ---- | --------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1    | No raw session token in D1.                                                             | Session schema and repositories use `token_hash`; `tests/repositories.test.ts` creates, reads, and revokes HMAC-backed sessions.                                                    |
| 2    | No raw email token in D1.                                                               | Verification-token schema and repositories use `token_hash`; route and repository tests consume magic, verification, and reset tokens through hashes.                               |
| 3    | Password hashes include algorithm, version, and parameters.                             | Password envelopes and parser live in `packages/core/src/index.ts`; `tests/crypto-password-session.test.ts` verifies envelope fields and invalid-password behavior.                 |
| 4    | Redirect targets are validated and stored at token creation.                            | `validateRedirectTarget` and token `redirect_to` storage are in core/worker code; route tests reject unsafe redirects and verify redirect allowlist separation.                     |
| 5    | Magic-login and email-verification GET links do not consume tokens.                     | `tests/routes.test.ts` verifies confirmation GET pages perform no D1 consume and POST consumes the token.                                                                           |
| 6    | Terminal email and dev outbox are development-only.                                     | `terminalEmail`, `byEnvironment`, and dev outbox checks are covered by `tests/email.test.ts` and `tests/routes.test.ts`.                                                            |
| 7    | Cloudflare Email is not required for local development.                                 | Generated config uses terminal email for development; quickstart docs and `pnpm smoke:wrangler-dev` exercise local auth without Cloudflare Email.                                   |
| 8    | Enumeration-sensitive request endpoints return generic responses with timing hardening. | Route tests cover generic magic/reset/verification responses, dummy token-HMAC work, and configured response delay.                                                                 |
| 9    | Preview/production emailed links use configured `publicOrigin`, not request `Host`.     | Runtime origin resolution and host validation live in `packages/worker/src/index.ts`; route and doctor tests cover missing public origin and production host rejection.             |
| 10   | Local HTTP does not use a `__Host-` cookie.                                             | Cookie resolution tests assert local cookies are unprefixed and production host-only cookies use `__Host-`.                                                                         |
| 11   | Auth routes own CORS/CSRF instead of relying on permissive global CORS.                 | Route tests cover allowed and disallowed preflights, missing origins, and unsupported credentialed cross-site CORS.                                                                 |
| 12   | Rate-limit keys and event hashes contain only derived email/IP/user-agent values.       | Core derivation helpers and route tests verify action/subject namespaces, opaque keys, and redacted auth events.                                                                    |
| 13   | Routes are not double-prefixed.                                                         | Hono adapter tests mount exactly one `/auth` prefix, and doctor tests detect double-prefixed auth routes.                                                                           |
| 14   | SQL uses prepared statements and bound parameters.                                      | D1 repositories use `prepare(...).bind(...)`; recovery helper tests verify identifiers are escaped before SQL construction and values are not interpolated.                         |
| 15   | `doctor` detects platform assumptions and deploy misconfiguration.                      | CLI tests cover missing D1, unapplied migrations, missing secrets, cookie hazards, email binding, Wrangler availability, package versions, and production source hazards.           |
| 16   | Request bodies are bounded before parsing.                                              | Worker body-limit handling and CLI body-limit diagnostics are covered by route and CLI tests.                                                                                       |
| 17   | Password hashing concurrency is bounded.                                                | `PasswordHashSemaphore` and runtime queue configuration are covered by crypto, route, security-hardening, and doctor tests.                                                         |
| 18   | Raw tokens are redacted from logs, errors, telemetry, and preview output.               | Shared redactor and route/email/CLI tests cover token URLs, token hashes, secrets, cookies, emails, IPs, and user agents.                                                           |
| 19   | Static config validation and request-time runtime resolution are separate.              | `defineAuthConfig` and runtime resolution are separate in `packages/worker/src/index.ts`; route and docs-coverage tests cover both surfaces.                                        |
| 20   | Unsupported cross-site cookie auth is rejected in v1.                                   | Config and route tests reject `SameSite=None` and unsupported credentialed cross-site CORS.                                                                                         |
| 21   | Password-reset GET renders a form and POST consumes.                                    | Route tests verify reset GET does not consume or hash passwords, while reset POST confirms and revokes old sessions.                                                                |
| 22   | Duplicate auth-secret key IDs are rejected.                                             | Key-ring parser tests and CLI rotate-secret tests reject duplicate current/previous `kid` values.                                                                                   |
| 23   | Production requests from untrusted hosts are rejected early.                            | Host validation tests reject untrusted production hosts before auth route handling.                                                                                                 |
| 24   | Production deploys require unambiguous Wrangler environment config.                     | CLI tests reject top-level development/ambiguous deploys and named environments missing production-safe vars or bindings.                                                           |
| 25   | Security decisions use primary-consistent D1 reads or guarded transactions.             | `createD1Repositories` uses `withSession("first-primary")` when available; repository tests cover primary reads and batched token/session state mutations.                          |
| 26   | Verification-token subject rows are valid for token type.                               | Core subject assertions, schema checks, and repository tests reject invalid magic, verification, and reset subject shapes.                                                          |
| 27   | Request-origin and redirect-origin allowlists are distinct.                             | Route tests specifically keep request CORS origins and post-auth redirect origins separate.                                                                                         |
| 28   | Repositories never generate raw auth tokens.                                            | Core repository contracts accept precomputed token hashes and session fields; repository tests use fixture HMAC envelopes and verify repositories do not return raw token material. |

## V1 Exclusion Audit

The v1 non-goals are tracked in `docs/non-goals.md`, `docs/roadmap.md`, and
`docs/known-limitations.md`. Source and docs searches should show these items
only as exclusions, generic route names, release evidence placeholders, or
third-party terms unrelated to implementation. The runtime does not implement
OAuth/social login, SAML, passkeys, MFA, organizations/teams, a
role/permission framework, hosted dashboards, a hosted auth service, billing
integration, admin impersonation, a multi-project control plane, or password
peppering.

## Release Rule

The project is locally ready for continued maintainer release work when the
local verification commands pass. It is not ready for public beta or stable 1.0
until every blocking evidence command above passes against real, redaction-safe
evidence and the pending approvals are replaced with dated, non-placeholder
signoffs.
